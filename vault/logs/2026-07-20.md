# 20/07/2026

## MCPs

Research on using MCP servers to let Hermes control homelab services (instead of blind shell commands or UI clicking).

### Principle: API-first apps use a purpose-built MCP, not Playwright

- **Playwright MCP** (`@playwright/mcp`, Microsoft) drives any web page via accessibility snapshots — but it is a _last resort_ for apps with no API. Fragile on UI changes, weak at reading terminal/canvas output (e.g. LXC console xterm), needs Node 18 + a real browser (~300-600MB RAM).
- If an app exposes a REST/WebSocket API, use its dedicated MCP server (or call the API via curl). More reliable, scoped permissions, no browser.

### Proxmox VE

- Playwright MCP _can_ drive the PVE web UI (login, click through nodes/VMs/LXCs) but is fragile and blind on the LXC console.
- **Better: `JerichoJack/Proxmox-MCP`** — purpose-built, talks to the PVE REST API + WebSocket events. Live state view, event ingestion (email/WebSocket), command tools, auto-remediation. No browser needed.
- Self-hosted Firecrawl (the earlier extraction option) does NOT fit current LXC: needs ~14GB RAM + Docker; current box is 4GB / 2-core / no Docker.

### Crafty Controller (Minecraft)

- Crafty 4 has a full v2 REST API (create/start/stop/backup servers, send stdin commands, manage files/users/schedules).
- **`HadiCherkaoui/crafty-mcp`** exposes the entire Crafty v2 API as MCP tools — natural-language MC server management. This is the right way to "set up a Minecraft server via Crafty" rather than telling Hermes to do it blind.
- Needs: Crafty 4 running, API token, MCP pointed at `http://<crafty-host>:<port>`.

### Home Assistant

- Has its own Assist API + REST/WebSocket API.
- Multiple **Home Assistant MCP servers** exist (e.g. `geekychris/homeassistant_mcp`, himcp.ai). Give tools to control/monitor devices and read state.
- Needs: Long-Lived Access Token + MCP pointed at HA LAN URL.

### How MCP setup works (general pattern)

An MCP server is a separate small program that exposes "tools" over the Model Context Protocol; Hermes launches/connects to it and calls those tools like its own. Three pieces every time:

1. **Install the server** (git clone + venv + `pip install -r requirements.txt`).
2. **Configure auth** — a `.env` or `env:` block with the target's credentials (PVE API token / Crafty v2 token / HA Long-Lived Access Token). Never commit live secrets to the vault.
3. **Register it in `~/.hermes/config.yaml`** under `mcp_servers`, then restart Hermes (tools are discovered at startup).

Two connection styles:

- **stdio**: Hermes launches it locally as a child process (`command` + `args`, optional `env:`). Runs inside Hermes's process tree — fits the 4GB LXC (no browser). Example:

  ```yaml
  mcp_servers:
    proxmox:
      command: "/path/to/Proxmox-MCP/.venv/bin/python"
      args: ["/path/to/Proxmox-MCP/mcp_server.py"]
      env:
        PROXMOX_HOST: "https://192.168.1.10:8006"
        PROXMOX_USER: "root@pam"
        PROXMOX_TOKEN_ID: "hermes@pam!mcp"
        PROXMOX_TOKEN_SECRET: "xxxxxxxx"
  ```

- **HTTP**: server runs as a web service (e.g. on the Proxmox host) and Hermes reaches it over LAN:

  ```yaml
  mcp_servers:
    proxmox:
      url: "http://192.168.1.10:8000/mcp"
  ```

  Per-server `tool_filter` lets you expose only the tools you want. Hermes also has a curated `hermes mcp` catalog of Nous-approved servers (n8n, Linear, GitHub…) installed one-click; Proxmox-MCP/crafty-mcp/HA are NOT in it, so they need a manual `config.yaml` entry.
